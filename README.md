# 리더십 딜레마 365: 위기의 12주

React + Vite 기반의 리더십 판단 시뮬레이션 MVP입니다. 현재 버전은 **localStorage 기반 mock storage**로 바로 실행되는 구조이며, 파일럿 리허설, 단일 브라우저 테스트, 강사용 데모 운영에 적합합니다.

이 앱은 정답 맞히기 게임이 아니라, 위기 상황에서 팀이 반복하는 판단 기준과 그 선택이 남기는 부담을 드러내는 **교육용 Decision Journey**입니다. 참가자는 팀 토의, 산출물 작성, 결과 카드 확인, 성찰을 거치며 “우리 팀은 위기 앞에서 무엇을 반복했는가”를 돌아봅니다.

---

## 1. 핵심 콘셉트

- 배경: 가상 B2B 기업 **세림인사이트**
- 구조: **Round 0 + Week 1~11 플레이 라운드 + Week 12 최종 판정**
- 목표: 조직개편 생존, 팀별 비밀 미션 달성, 판단 습관 성찰
- 방식: 개인 생각 → 팀 결정 → 산출물 → 상태값 변화 → 결과 카드 → 최종 성찰
- 운영 철학: 점수 경쟁보다 선택의 기준, 대가, 반복 패턴을 보는 교육용 여정

---

## 2. 포함 기능

### 기본 진행

- Host 방 생성 및 입장 코드 생성
- 6개 팀 운영: 영업, 마케팅, 연구개발, 운영, HR, 재무
- 기본 대면 운영: **팀당 1개 Team 화면**
- 확장 운영: Player 개인 선택·성찰 화면 선택 사용
- Round 0 KSA 선택
- Week 1~11 팀 최종 선택, 토의 요약, 산출물 입력
- Week 12 개인 성찰, 팀 선언문, 최종 판정 생성

### 계산 모델 v2

- 토의 요약 품질 리뷰
- 산출물 증거성 리뷰
- 산출물 품질 v2 계산
- 라운드 영향도 TOP 3 로그
- 표시 리스크와 누적 리스크 분리
- 누적 리스크 흐름 저장
- 최종 게이트 판정
- 팀 선언문 품질 피드백
- 개인 성찰 품질 피드백

### 강사용·운영자 기능

- Host Dashboard 진행 제어
- 강사용 설명 패널
- 파일럿 운영용 최종 사용 가이드
- 파일럿 직전 기능 동선 점검표
- 교육 리포트와 Markdown 다운로드
- 운영 QA 점검판
- 리허설 샘플 시나리오 4종
- 리허설 샘플 밸런스 검증
- 계산 모델 v2 데이터 보정
- 전체 재계산 + 최종 판정 재생성
- JSON 백업과 JSON 가져오기
- 현재 방 진행 데이터 초기화

## 3. 실행 방법

```bash
npm install
npm run dev
```

브라우저에서 Vite가 표시하는 로컬 주소로 접속합니다.

빌드 확인은 다음 명령으로 실행합니다.

```bash
npm run build
```

## 4. 주요 라우트

| 라우트 | 역할 |
|---|---|
| `/` | 랜딩 화면 |
| `/host/create` | Host 방 생성 |
| `/host/:roomId` | Host Dashboard / 진행 제어 |
| `/join/:joinCode` | 확장 운영용 참가자 입장 |
| `/player/:roomId/:playerId` | Player 개인 선택·성찰 화면 |
| `/team/:roomId/:teamId` | Team KSA·팀 결정·산출물·선언문 화면 |
| `/competencies/:roomId` | 팀원 역량 프로필 / 인물 카드 구성 |
| `/compare/:roomId` | 다팀 비교 |
| `/report/:roomId` | 교육 리포트 |
| `/guide/:roomId` | 강사용 진행 가이드 |
| `/admin/:roomId` | 운영 QA 점검판 / 리허설 샘플 / 백업 |

`vercel.json`이 포함되어 있어 직접 URL 접속과 새로고침도 `index.html`로 연결됩니다.

## 5. 기본 운영 흐름

1. `/host/create`에서 방을 생성합니다.
2. Host 화면에서 팀 화면 링크를 준비합니다.
3. Round 0에서 팀별 KSA 9개를 저장합니다.
4. Week 1~11에서 상황 읽기 → 1분 개인 생각 → 팀 토의 → 팀 최종 선택 저장 → 산출물 저장을 반복합니다.
5. Host 화면에서 **결과 계산 후 공개**를 실행합니다.
6. 팀 화면에서 결과 카드를 확인하고 다음 Week로 이동합니다.
7. Week 12에서 팀 선언문을 저장합니다.
8. Host 화면에서 최종 판정을 생성합니다.
9. `/report/:roomId`에서 리포트와 Markdown을 저장합니다.
10. `/admin/:roomId`에서 QA 리포트와 JSON 백업을 저장합니다.

---

## 6. 파일럿 운영 전 체크

파일럿 전에는 다음 순서로 점검합니다.

1. `docs/pilot-document-package-index.md`를 확인합니다.
2. `docs/pilot-operator-package.md`를 확인합니다.
3. `/guide/:roomId`의 **파일럿 직전 기능 동선 점검표**에서 한 팀 기준 P1~P9를 실행합니다.
4. `/admin/:roomId`에서 리허설 샘플 1개를 생성합니다.
5. 리허설 밸런스 검증을 확인합니다.
6. `/report/:roomId`에서 Markdown 다운로드를 확인합니다.
7. Admin에서 JSON 백업을 다운로드합니다.
8. 실제 운영 전 현재 방 진행 데이터를 초기화합니다.

## 7. 리허설 샘플 시나리오

Admin 화면에서 실제 파일럿 전에 리허설 데이터를 바로 생성할 수 있습니다. 버튼을 누르면 기존 리허설 데이터가 정리되고, 시나리오에 맞는 선택 패턴, 산출물 품질, 인물 카드 구성, 성찰, 선언문, 최종 판정이 새로 생성됩니다.

| 시나리오 | 용도 |
|---|---|
| 미션 성공형 | 정상 운영, 생존/미션 달성 흐름 확인 |
| 에이스 과부하형 | 빠른 실행의 성과와 에이스 소진 부담 디브리핑 |
| 루머 대응형 | 불안·신뢰·협업 부담 디브리핑 |
| 재편 위험형 | 낮은 산출물과 반복 속도 선택의 대가 확인 |

샘플 생성 후 확인할 화면은 `/admin/:roomId`, `/guide/:roomId`, `/report/:roomId`, `/competencies/:roomId`입니다.

## 8. 참가자에게 노출하지 않을 정보

아래 정보는 참가자에게 산식이나 기준표처럼 설명하지 않습니다.

- 팀별 비밀 미션 세부 기준
- 최종 게이트 내부 판단 기준
- 산출물 품질 계산 비중
- 누적 리스크 원점수
- 인물 카드별 리스크 보정 방식
- 라운드 영향도 TOP 3 계산 기준

강사는 이를 다음 질문으로 바꿔 사용합니다.

- 우리 팀은 어떤 기준을 반복했습니까?
- 이 선택은 무엇을 얻고 무엇을 남겼습니까?
- 다음 현업에서 먼저 낮춰야 할 부담은 무엇입니까?
- 팀 선언문은 다음 회의 행동으로 이어질 만큼 구체적입니까?

## 10. 현재 제한과 다음 개발 우선순위

### 현재 제한

- 현재 데이터 저장은 localStorage 기반입니다.
- 여러 기기 간 실시간 공유는 아직 지원하지 않습니다.
- 의존성은 아직 `latest` 기반이며, 파일럿 빌드 검증 후 lock 파일과 버전 고정을 진행합니다.
- 실제 파일럿 전에는 `npm run build`와 배포 상태를 별도로 확인해야 합니다.

### 다음 개발 우선순위

1. 실제 CI 빌드 결과 확인과 오류 수정
2. `package-lock.json` 생성 및 `npm ci` 전환
3. Firestore adapter 구현
4. Host 권한/PIN 구조
5. 실제 다중 접속 파일럿
6. 리포트 PDF/docx 출력 강화
