# 리허설 샘플 밸런스 검증 기준

이 문서는 Admin 화면의 리허설 샘플 4종이 의도한 결과 분포를 만드는지 확인하기 위한 기준입니다.

전체 Week가 플레이 라운드로 확장되면서 리허설 샘플도 Week 1~11 전체 선택과 산출물을 생성합니다.

---

## 1. 검증 목적

리허설 샘플은 실제 교육 운영 전에 다음을 확인하기 위한 데이터입니다.

- 최종 판정이 한쪽으로 과도하게 쏠리지 않는가?
- 미션 성공형은 실제로 긍정 판정이 나오는가?
- 에이스 과부하형은 성과와 부담이 함께 드러나는가?
- 루머 대응형은 불안 신호와 회복 가능성이 함께 보이는가?
- 재편 위험형은 재편 신호가 나오되 모든 팀이 무조건 탈락하지는 않는가?

---

## 2. 샘플별 기대 결과

| 샘플 | 기대 결과 |
|---|---|
| 미션 성공형 | 대부분의 팀이 조건부 유지 이상, 미션 부분 달성 이상 |
| 에이스 과부하형 | 에이스소진위험 또는 실행압박이 남고, 일부 팀은 조건부/재편 신호 |
| 루머 대응형 | 불안·신뢰·협업 부담 중 하나가 남되, 일부 팀은 생존 가능 |
| 재편 위험형 | 최소 1개 팀에서 재편 신호가 나오지만 모든 팀이 탈락하지는 않음 |

---

## 3. 자동 검증 유틸

`src/utils/rehearsalBalanceUtils.js`에 `buildRehearsalBalanceCheck(room)`을 추가했습니다.

이 함수는 다음을 반환합니다.

```text
scenarioKey
scenarioLabel
teamCount
finalCount
passedCount
totalChecks
status
checks
summary
```

`status`는 모든 기준을 충족하면 `정상`, 하나라도 어긋나면 `점검 필요`입니다.

---

## 4. 시나리오 감지 방식

Admin 리허설 샘플은 참가자 ID에 시나리오 키를 포함합니다.

예시:

```text
sales_success_1
hr_ace_2
rnd_rumor_3
finance_pressure_1
```

검증 유틸은 이 값을 보고 현재 방이 어떤 샘플인지 감지합니다.

---

## 5. 운영자가 확인할 것

리허설 샘플 생성 후 다음 순서로 확인합니다.

1. Admin 화면에서 샘플 생성
2. 최종 판정 생성 여부 확인
3. `/report/:roomId`에서 팀별 결과 확인
4. 기대 결과와 실제 결과 비교
5. 모든 팀이 같은 결과로 쏠리면 밸런싱 기준 재검토

---

## 6. 조정 후보

검증 결과가 어긋나면 다음 순서로 조정합니다.

1. 리허설 샘플의 `choiceTypes`
2. 리허설 샘플의 `qualities`
3. 선택지 `baseEffects`
4. 인물 카드 영향 완충 기준
5. 최종 판정 threshold
6. 비밀 미션 기준

---

## 7. 파일럿 후 판단 기준

실제 파일럿 후에는 자동 검증보다 교육 효과를 우선합니다.

중요한 기준은 다음입니다.

- 참가자가 결과 차이를 납득했는가?
- 팀별 선택 패턴이 리포트에서 구분되는가?
- 강사가 디브리핑할 만한 차이가 생겼는가?
- 조건부 생존 팀이 적절히 발생하는가?
- 탈락/재편 판정이 교육적으로 지나치게 가혹하지 않은가?
