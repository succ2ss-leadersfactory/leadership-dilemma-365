# v1.0 Pilot Release Notes

릴리즈명: **리더십 딜레마 365: 위기의 12주 — v1.0 Pilot**

이 문서는 파일럿 운영 전 공유용 릴리즈 요약입니다.

---

## 1. 이번 버전의 목적

v1.0 Pilot은 교육장 리허설과 단일 브라우저 데모 운영을 위한 첫 운영 버전입니다.

핵심 목적은 다음과 같습니다.

- 12주 위기 상황을 따라가며 팀의 판단 습관을 확인합니다.
- 개인 판단, 팀 결정, 산출물, 결과 카드, 성찰 흐름을 연결합니다.
- 팀원 인물 카드와 역량 프로필을 활용해 디브리핑 소재를 제공합니다.
- 선택의 대가, 산출물 품질, 누적 리스크, 최종 게이트를 강사용으로 해석합니다.
- 교육생 화면은 피드백과 토의 흐름 중심으로, 강사용 화면은 판정 근거와 개입 질문 중심으로 분리합니다.

---

## 2. 포함 기능

### 참가자·팀 진행

- Host 방 생성과 입장 코드
- 팀당 1개 화면 중심의 기본 대면 운영
- 온라인/확장 운영용 Player 화면
- Round 0 KSA 선택
- 팀원 역량 프로필 자동 생성
- 팀원 인물 카드 자동 배정
- Week 1~11 플레이 라운드
- Week 12 개인 성찰, 팀 선언문, 최종 판정

### 계산 모델 v2

- 토의 요약 품질 리뷰
- 산출물 증거성 리뷰
- 산출물 품질 v2 계산
- 라운드 영향도 TOP 3 로그
- 표시 리스크와 누적 리스크 분리
- 누적 리스크 흐름 저장
- 최종 게이트 판정
- 팀 선언문 품질 피드백
- 개인 성찰 품질 피드백

### 강사용·운영자 기능

- Host Dashboard 진행 제어
- 강사용 설명 패널
- 강사용 최종 운영 가이드
- 파일럿 직전 기능 동선 점검표
- 교육 리포트와 Markdown 다운로드
- 운영 QA 점검판
- 리허설 샘플 시나리오 4종
- 리허설 샘플 밸런스 검증
- 계산 모델 v2 데이터 보정
- 전체 재계산 + 최종 판정 재생성
- JSON 백업과 JSON 가져오기
- 현재 방 진행 데이터 초기화

## 3. 주요 화면

| 화면 | 경로 | 역할 |
|---|---|---|
| 방 생성 | `/host/create` | 새 파일럿 방 생성 |
| Host Dashboard | `/host/:roomId` | 단계 이동, 결과 공개, 최종 판정 생성 |
| Team | `/team/:roomId/:teamId` | KSA, 팀 결정, 산출물, 결과 카드, 팀 선언문 |
| Player | `/player/:roomId/:playerId` | 선택 운영용 개인 판단·성찰 입력 |
| 강사 가이드 | `/guide/:roomId` | 최종 운영 가이드, 동선 점검표, 라운드 질문 |
| 운영 QA | `/admin/:roomId` | 리허설, 보정, 재계산, 초기화, QA 리포트 |
| 교육 리포트 | `/report/:roomId` | 최종 판정, 누적 리스크, 성찰·선언 피드백 |
| 역량 프로필 | `/competencies/:roomId` | 팀원 역량과 인물 카드 |
| 다팀 비교 | `/compare/:roomId` | 팀별 판단 패턴과 리스크 비교 |

## 4. 파일럿 전 확인 순서

1. `README.md`로 실행 방법과 주요 route를 확인합니다.
2. `docs/pilot-document-package-index.md`로 운영 문서 위치를 확인합니다.
3. `docs/pilot-operator-package.md`로 당일 운영 순서를 확인합니다.
4. `npm install` 후 `npm run dev`로 로컬 실행을 확인합니다.
5. `/host/create`에서 새 방을 만듭니다.
6. `/guide/:roomId`에서 파일럿 직전 기능 동선 점검표 P1~P9를 한 팀 기준으로 확인합니다.
7. `/admin/:roomId`에서 리허설 샘플 1개를 생성합니다.
8. `/admin/:roomId`에서 리허설 밸런스 검증을 확인합니다.
9. `/report/:roomId`에서 Markdown 다운로드를 확인합니다.
10. Admin에서 JSON 백업을 다운로드합니다.
11. 실제 운영 전 현재 방 진행 데이터를 초기화합니다.

## 5. 리허설 샘플

Admin 화면에서 다음 샘플을 생성할 수 있습니다.

| 샘플 | 목적 |
|---|---|
| 미션 성공형 | 정상 운영, 생존/미션 달성 흐름 확인 |
| 에이스 과부하형 | 빠른 실행의 성과와 에이스 소진 부담 디브리핑 |
| 루머 대응형 | 불안·신뢰·협업 부담 디브리핑 |
| 재편 위험형 | 낮은 산출물과 반복 속도 선택의 대가 확인 |

샘플 생성 후 반드시 확인할 화면은 다음입니다.

- `/admin/:roomId`: 리허설 밸런스 검증
- `/guide/:roomId`: 강사용 질문과 운영 가이드
- `/report/:roomId`: 교육 리포트와 Markdown 다운로드
- `/competencies/:roomId`: 인물 카드와 역량 프로필

## 6. 교육생/강사용 정보 분리

교육생 화면은 다음 정보 중심입니다.

```text
현재 할 일
개인 판단과 이유
팀 토의 출발점
산출물 작성 기준
작은 진전
남은 부담
다음 행동
성찰 질문
```

강사용 화면과 리포트에는 다음 정보가 유지됩니다.

```text
비밀 미션 제목과 점수
판정 근거
최종 게이트 체크
중간 사건 후폭풍 근거
누적 리스크 흐름
영향도 TOP 3
인물 카드 영향 기록
팀별 전문성 경고 신호
산출물 증거 수준
역량 프로필 상세 수준
```

참가자에게 내부 산식이나 판정 기준표처럼 설명하지 않습니다.

## 7. 운영 메모

- 참가자에게 먼저 “정답 맞히기 게임이 아니다”라고 안내합니다.
- 기본 대면 운영은 팀당 1개 화면으로 진행합니다.
- 개인 화면은 온라인·확장 운영에서만 선택적으로 사용합니다.
- 팀 토의에서는 다수결보다 선택이 갈린 이유를 먼저 봅니다.
- 산출물은 긴 글이 아니라 실행 약속으로 작성하게 합니다.
- 결과 카드에서는 숫자보다 남은 부담을 먼저 봅니다.
- 인물 카드는 평가가 아니라 토의용 역할 렌즈로 다룹니다.
- 마지막에는 개인 성찰과 팀 선언문을 다음 현업 행동으로 연결합니다.
- 초기화 전에는 JSON 백업을 먼저 다운로드합니다.

---

## 8. 현재 제한

- 현재 데이터 저장은 localStorage 기반입니다.
- 여러 기기 간 실시간 공유는 후속 버전에서 다룹니다.
- Firestore 전환은 별도 Sprint에서 진행합니다.
- 의존성은 현재 `latest` 기반이며, 파일럿 빌드 검증 후 lock 파일과 버전 고정을 진행합니다.
- 로컬 파일럿 전에는 `npm run build`를 별도로 실행해 확인해야 합니다.
