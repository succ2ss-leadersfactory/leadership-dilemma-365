# 파일럿 운영자 인수인계 가이드

이 문서는 **리더십 딜레마 365: 위기의 12주**를 파일럿으로 운영할 사람에게 넘기는 최종 안내서입니다.

목표는 운영자가 “어디서 무엇을 보면 되는가”를 빠르게 파악하는 것입니다.

## 2. 운영 전 필수 확인

운영 전에는 아래 순서만 확인하면 됩니다.

1. `docs/facilitator-pilot-runbook.md`로 당일 진행 순서를 확인합니다.
2. `docs/pilot-ui-ux-qa-checklist.md`로 교육생/강사용 화면 노출 기준을 확인합니다.
3. `npm install`
4. `npm run build`
5. `/host/create`에서 방 생성
6. `/admin/:roomId` 접속
7. 리허설 샘플 1개 생성
8. 리허설 샘플 밸런스 검증 패널 확인
9. QA 리포트 다운로드
10. `/report/:roomId`에서 교육 리포트 다운로드
11. `docs/pilot-final-qa-decision-table.md` 기준으로 운영 가능 여부 판정
12. 운영 전 JSON 백업을 저장합니다.

## 3. 실제 운영 화면 순서

| 단계 | 화면 | 확인할 것 |
|---|---|---|
| 방 생성 | `/host/create` | 입장 코드 생성 |
| 진행 제어 | `/host/:roomId` | 현재 라운드와 단계 |
| 참가자 입장 | `/join/:joinCode` | 이름, 팀 선택 |
| 개인 판단 | `/player/:roomId/:playerId` | 선택과 이유 저장 |
| 팀 결정 | `/team/:roomId/:teamId` | 팀 최종 선택, 산출물 저장 |
| 운영 점검 | `/admin/:roomId` | 누락 데이터, 샘플, QA |
| 강사 가이드 | `/guide/:roomId` | 진행 멘트, 전문성 렌즈, 개입 질문 |
| 리포트 | `/report/:roomId` | 최종 판정, 성찰, 전문성 요약, 다운로드 |

## 4. 운영자가 기억할 메시지

참가자에게 처음에 이렇게 안내합니다.

> 이 게임은 정답을 맞히는 게임이 아닙니다. 위기 상황에서 우리 팀이 어떤 기준으로 판단하고, 그 선택이 어떤 부담을 남기는지 확인하는 과정입니다.

개인 판단 단계에서는 이렇게 말합니다.

> 지금은 팀 토의 전에 각자 먼저 판단하는 시간입니다. 다른 사람 의견을 보기 전에 내가 본 핵심 문제와 선택 이유를 남겨 주세요.

팀 토의 단계에서는 이렇게 말합니다.

> 다수 의견을 바로 고르지 말고, 왜 의견이 갈렸는지 먼저 이야기해 주세요. 팀으로 감수할 선택을 정하는 것이 목표입니다.

결과가 낮은 팀에게는 이렇게 말합니다.

> 이 결과는 실패 판정이 아니라, 현업에서 먼저 다뤄야 할 부담을 보여주는 신호입니다.

전문성 신호가 낮은 팀에게는 이렇게 말합니다.

> 산출물 점수가 낮다는 뜻이 아니라, 상무나 타부서가 믿을 수 있는 증거가 아직 부족하다는 뜻입니다.

## 5. 교육생/강사용 정보 분리 기준

운영자는 아래 기준을 기억합니다.

교육생 화면에는 다음이 보여야 합니다.

```text
현재 할 일
개인 판단과 이유
팀 토의 출발점
산출물 작성 기준
작은 진전
남은 부담
다음 행동
성찰 질문
```

교육생 화면에 과도하게 보이면 안 되는 정보는 다음입니다.

```text
강사용 질문
비밀 미션 점수
세부 판정 로직
내부 증거 점수
상세 판정 근거 전체
```

강사용 리포트에는 다음이 유지되어야 합니다.

```text
비밀 미션 제목과 점수
비밀 미션 근거
종합 판정 근거
중간 사건 후폭풍 근거
강사용 질문
인물 카드 영향 기록
팀별 전문성 경고 신호
산출물 증거 수준
```

## 6. 리허설 샘플 사용법

Admin 화면에서 4개 샘플을 만들 수 있습니다.

| 샘플 | 용도 |
|---|---|
| 미션 성공형 | 정상 운영과 리포트 확인 |
| 에이스 과부하형 | 빠른 실행과 소진 리스크 디브리핑 |
| 루머 대응형 | 불안·신뢰·협업 부담 디브리핑 |
| 재편 위험형 | 낮은 산출물과 재편 신호 확인 |

샘플 생성 후에는 같은 Admin 화면의 **리허설 샘플 밸런스 검증** 패널을 확인합니다.

상태가 `점검 필요`이면 선택 패턴, 산출물 품질, 선택지 효과, 최종 판정 기준 순서로 확인합니다.

## 7. 전문성 고도화 확인

전문성 고도화 구조는 `docs/expertise-index.md`에서 먼저 확인합니다.

| 화면 | 확인할 것 |
|---|---|
| `/team/:roomId/:teamId` | 산출물 입력 기준과 증거 수준 |
| `/guide/:roomId` | 팀별 전문성 렌즈와 강사용 질문 |
| `/report/:roomId` | 팀별 전문성 요약과 누적 경고 신호 |
| `/admin/:roomId` | 전문성 렌즈, 증거 수준, 결과 해석 QA |

## 8. 운영 중 막히는 경우

| 상황 | 먼저 확인할 것 |
|---|---|
| 참가자가 안 보임 | 참가자 입장 코드 확인 |
| 참가자가 Player 화면을 잃어버림 | 개인 링크를 다시 전달 |
| 개인 판단이 팀 화면에 안 보임 | 해당 참가자의 개인 판단 저장 여부 확인 |
| 결과가 안 나옴 | 팀 결정과 산출물 저장 여부 확인 |
| 최종 판정이 안 보임 | Week 12 최종 판정 생성 여부 확인 |
| 전문성 신호가 안 보임 | Admin QA에서 evidenceReview 생성 여부 확인 |
| 리포트가 비어 있음 | 리허설 샘플 생성 또는 전체 라운드 계산 여부 확인 |
| 저장 상태가 불안함 | JSON 백업을 먼저 다운로드 |


## 10. 현재 제한과 다음 우선순위

- 현재 저장소는 localStorage 기반입니다.
- 여러 기기 간 실시간 공유는 아직 지원하지 않습니다.
- 실제 파일럿 전 `npm run build`와 배포 확인이 필요합니다.
- 전문성 증거 수준은 자동 키워드 기반 보조 판단입니다.
- 다음 우선순위는 실제 CI 빌드 확인, lock 파일 생성, Markdown 리포트 전문성 항목 확장입니다.
