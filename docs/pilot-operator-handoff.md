# 파일럿 운영자 인수인계 가이드

이 문서는 **리더십 딜레마 365: 위기의 12주**를 파일럿으로 운영할 사람에게 넘기는 최종 안내서입니다.

목표는 운영자가 “어디서 무엇을 보면 되는가”를 빠르게 파악하는 것입니다.

## 2. 운영 전 필수 확인

운영 전에는 아래 순서만 확인하면 됩니다.

1. `npm install`
2. `npm run build`
3. `/host/create`에서 방 생성
4. `/admin/:roomId` 접속
5. 리허설 샘플 1개 생성
6. 리허설 샘플 밸런스 검증 패널 확인
7. QA 리포트 다운로드
8. `/report/:roomId`에서 교육 리포트 다운로드
9. `docs/pilot-final-qa-decision-table.md` 기준으로 운영 가능 여부 판정

## 3. 실제 운영 화면 순서

| 단계 | 화면 | 확인할 것 |
|---|---|---|
| 방 생성 | `/host/create` | 입장 코드 생성 |
| 진행 제어 | `/host/:roomId` | 현재 라운드와 단계 |
| 참가자 입장 | `/join/:joinCode` | 이름, 팀 선택 |
| 개인 선택 | `/player/:roomId/:playerId` | 선택과 이유 저장 |
| 팀 결정 | `/team/:roomId/:teamId` | 팀 최종 선택, 산출물 저장 |
| 운영 점검 | `/admin/:roomId` | 누락 데이터, 샘플, QA |
| 강사 가이드 | `/guide/:roomId` | 진행 멘트, 개입 질문 |
| 리포트 | `/report/:roomId` | 최종 판정, 성찰, 다운로드 |

## 4. 운영자가 기억할 메시지

참가자에게 처음에 이렇게 안내합니다.

> 이 게임은 정답을 맞히는 게임이 아닙니다. 위기 상황에서 우리 팀이 어떤 기준으로 판단하고, 그 선택이 어떤 부담을 남기는지 확인하는 과정입니다.

결과가 낮은 팀에게는 이렇게 말합니다.

> 이 결과는 실패 판정이 아니라, 현업에서 먼저 다뤄야 할 부담을 보여주는 신호입니다.

---

## 5. 리허설 샘플 사용법

Admin 화면에서 4개 샘플을 만들 수 있습니다.

| 샘플 | 용도 |
|---|---|
| 미션 성공형 | 정상 운영과 리포트 확인 |
| 에이스 과부하형 | 빠른 실행과 소진 리스크 디브리핑 |
| 루머 대응형 | 불안·신뢰·협업 부담 디브리핑 |
| 재편 위험형 | 낮은 산출물과 재편 신호 확인 |

샘플 생성 후에는 같은 Admin 화면의 **리허설 샘플 밸런스 검증** 패널을 확인합니다.

상태가 `점검 필요`이면 선택 패턴, 산출물 품질, 선택지 효과, 최종 판정 기준 순서로 확인합니다.

---

## 6. 운영 중 막히는 경우

| 증상 | 먼저 볼 곳 |
|---|---|
| 참가자가 안 보임 | `/admin/:roomId`의 참가자 수 |
| KSA가 안 생김 | Team 화면의 Round 0 저장 여부 |
| 결과 카드가 안 보임 | 팀 결정과 산출물 저장 여부 |
| 최종 판정이 안 나옴 | Week 12 최종 판정 생성 여부 |
| 리포트가 비어 있음 | 최종 판정, 팀 선언문, 개인 성찰 여부 |
| 샘플 결과가 이상함 | 리허설 샘플 밸런스 검증 패널 |

---

## 7. 현재 제한

현재 버전은 localStorage 기반입니다.

- 같은 브라우저에서는 데이터가 유지됩니다.
- 여러 기기 간 실시간 공유는 아직 지원하지 않습니다.
- 실제 다중 접속 운영은 Firestore 전환 후 안정화됩니다.
- 의존성은 아직 `latest` 기반이므로 파일럿 안정화 후 lock 파일을 생성합니다.
