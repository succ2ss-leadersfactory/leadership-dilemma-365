# 플레이 라운드와 LOG 후폭풍 처리 기준

Week 2, 3, 4, 6, 7, 9가 실제 플레이 라운드로 확장되면서 기존 중간 LOG와 역할이 겹칠 수 있습니다.

이 문서는 v1.0 Pilot 이후의 처리 기준을 정리합니다.

---

## 1. 기본 원칙

같은 Week가 플레이 라운드로 존재하고, 해당 팀의 팀 결정이 저장되어 있으면 그 Week의 LOG 후폭풍은 최종 판정에서 별도로 다시 반영하지 않습니다.

즉, 같은 사건은 다음 둘 중 하나로만 최종 판정에 영향을 줍니다.

```text
플레이 라운드 선택·산출물·계산 결과
또는
LOG 후폭풍 규칙
```

둘을 동시에 반영하지 않습니다.

---

## 2. 이유

Week 2, 3, 4, 6, 7, 9는 원래 중간 사건 LOG였습니다.

이후 전체 Week 플레이 구조로 확장되면서 해당 Week는 다음 데이터를 갖게 되었습니다.

- 팀 선택
- 팀 결정
- 산출물
- 결과 계산
- 역량 성장/위축
- 인물 카드 영향

따라서 같은 사건을 LOG 후폭풍으로 다시 더하면 리스크가 이중 계산될 수 있습니다.

---

## 3. 현재 구현 기준

`calculateWeekLogImpacts()`는 기본값으로 `skipPlayableLogs = true`를 사용합니다.

팀 결정에 같은 Week의 플레이 라운드가 있으면 해당 LOG는 건너뜁니다.

예시:

```text
log_week3_chat_spread
week: 3

팀 결정에 week3이 있으면
→ log_week3_chat_spread 후폭풍은 별도 적용하지 않음
```

---

## 4. 최종 판정 기록

건너뛴 LOG는 사라지는 것이 아니라 최종 판정 데이터에 기록됩니다.

저장 필드:

```text
skippedPlayableLogCount
skippedPlayableLogs
```

최종 판정 근거에는 다음 의미의 문장이 남습니다.

```text
중간 사건은 플레이 라운드 선택과 산출물로 이미 반영되어 별도 LOG 후폭풍에서는 제외했습니다.
```

---

## 5. 강사용 해석

강사는 다음처럼 설명하면 됩니다.

> 이 사건은 별도의 후폭풍 점수로 다시 더하지 않았습니다. 이미 해당 Week에서 여러분이 선택하고 산출물로 다룬 결과가 상태값에 반영되어 있습니다.

---

## 6. 후속 개선 후보

향후에는 LOG를 두 종류로 나누는 것이 좋습니다.

| 구분 | 설명 |
|---|---|
| Playable Event | 직접 선택과 산출물이 있는 Week 사건 |
| Passive Log | 선택 라운드 사이에 누적되는 배경 사건 |

이렇게 분리하면 타임라인과 최종 판정 설명이 더 명확해집니다.
